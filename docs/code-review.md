# Code Review 指南

本文档说明代码审查的流程、标准和最佳实践。

---

## Code Review 的目的

- 发现代码中的 bug 和潜在问题
- 确保代码符合项目规范
- 知识共享和团队学习
- 提高代码质量和可维护性

---

## Review 检查清单

### 功能正确性

- [ ] 代码实现了需求描述的功能
- [ ] 边界情况得到正确处理
- [ ] 错误处理完善且恰当
- [ ] 没有引入新的 bug

### 测试覆盖

- [ ] 有对应的单元测试
- [ ] 测试覆盖了正常和异常情况
- [ ] 测试覆盖率 ≥ 80%
- [ ] 测试描述清晰，易于维护

### 代码质量

- [ ] 命名清晰、描述性强
- [ ] 函数职责单一，长度适中（<50 行）
- [ ] 没有重复代码（DRY 原则）
- [ ] 遵循项目代码规范
- [ ] 没有硬编码的配置值

### 架构和设计

- [ ] 模块职责清晰
- [ ] 接口设计合理
- [ ] 依赖关系正确
- [ ] 符合 SOLID 原则

### 安全性

- [ ] 没有敏感信息泄露（密钥、密码等）
- [ ] 输入验证完善
- [ ] 没有 SQL 注入、XSS 等漏洞
- [ ] 权限检查正确

### 性能

- [ ] 没有明显的性能问题
- [ ] 数据库查询优化合理
- [ ] 没有不必要的循环或计算
- [ ] 大数据量有分页/流式处理

### 文档

- [ ] 复杂逻辑有注释说明
- [ ] 公共 API 有文档注释
- [ ] README/CHANGELOG 更新（如需要）

---

## Review 流程

### 1. 提交前自检

**作者**在提交 PR 前应该：

```bash
# 1. 运行测试
npm test
pytest

# 2. 检查覆盖率
npm run test:coverage
pytest --cov

# 3. 运行 linter
npm run lint
black . --check
ruff check .

# 4. 确认所有检查通过
```

### 2. 创建 PR

**PR 标题格式**：遵循 Conventional Commits
```
feat: 添加用户认证功能
fix: 修复数据库连接泄漏
```

**PR 描述模板**：
```markdown
## 变更类型
- [ ] feat 新功能
- [ ] fix 修复 bug
- [ ] refactor 重构
- [ ] docs 文档
- [ ] test 测试
- [ ] chore 构建/工具

## 变更说明
简要描述做了什么

## 测试情况
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] E2E 测试通过
- [ ] 测试覆盖率 ≥ 80%

## 检查清单
- [ ] 代码符合项目规范
- [ ] 没有硬编码敏感信息
- [ ] 文档已更新（如需要）

## 相关 Issue
Closes #123
```

### 3. Review 过程

**Reviewer** 检查：

1. **整体把握**（5 分钟）
   - 理解变更的目的
   - 查看整体架构影响
   - 确认变更范围合理

2. **详细审查**（15-30 分钟）
   - 逐文件检查
   - 关注关键逻辑
   - 标记需要修改的地方

3. **反馈意见**
   - 使用建设性的语言
   - 解释"为什么"而不仅仅是"是什么"
   - 区分"必须修改"和"建议优化"

### 4. 反馈分类

| 类型 | 标记 | 说明 | 必须修改 |
|------|------|------|----------|
| 阻断问题 | `🚫 Must fix` | Bug、安全漏洞、严重规范问题 | ✅ |
| 重要建议 | `⚠️ Should fix` | 潜在问题、规范改进 | ✅ |
| 优化建议 | `💡 Consider` | 代码改进、最佳实践 | ❌ |
| 好的实践 | `👍 Well done` | 值得学习的代码 | ❌ |

### 5. 反馈示例

```markdown
# 好的反馈

## 🚫 Must fix: 安全问题

第 45 行：直接拼接 SQL 查询存在注入风险
```typescript
const query = `SELECT * FROM users WHERE id = ${userId}`; // ❌
```

建议使用参数化查询：
```typescript
const query = 'SELECT * FROM users WHERE id = $1'; // ✅
await db.query(query, [userId]);
```

---

## ⚠️ Should fix: 错误处理

第 78 行：catch 块没有处理错误
```typescript
try {
  await saveData(data);
} catch (error) {
  // 什么都不做 ⚠️
}
```

建议至少记录日志或向上抛出。

---

## 💡 Consider: 函数命名

函数名 `process()` 不够具体，建议改为 `processUserData()` 更清晰。
```

## Review 轮次

### 第一轮：核心问题

- 关注：功能正确性、安全性、严重规范问题
- 不纠结：命名风格、空行、注释等

### 第二轮：改进建议

- 关注：代码质量、性能、可维护性
- 提出：优化建议、最佳实践

### 第三轮：细节完善（可选）

- 关注：命名、格式、文档
- 目标：达到"可以合并"的标准

---

## 通过标准

PR 可以合并的条件：

- [ ] 所有 `🚫 Must fix` 问题已解决
- [ ] 所有 `⚠️ Should fix` 问题已解决或给出合理解释
- [ ] 测试全部通过
- [ ] CI/CD 检查通过
- [ ] 至少一人 Approve

## 常见问题

### 1. Review 太慢怎么办？

**优先级排序**：
- 安全/正确性问题 > 架构问题 > 代码风格
- 核心路径 > 边缘场景
- 公共 API > 内部实现

**时间控制**：
- 小 PR（<200 行）：30 分钟内
- 中 PR（200-500 行）：1 小时内
- 大 PR（>500 行）：分批审查或要求拆分

### 2. 意见分歧怎么办？

**讨论原则**：
- 基于数据和事实，而非个人偏好
- 参考项目的编码规范
- 当无法达成一致时，由项目维护者决定

**友好提醒**：
```markdown
@author 我们对这个设计有不同看法。能否约个时间
讨论一下？我觉得当面/语音交流可能更高效。
```

### 3. 如何处理大 PR？

**建议拆分**：
```markdown
@author 这个 PR 变更较大（500+ 行），建议拆分成多个小 PR：
1. 第一个 PR：添加数据模型和基础接口
2. 第二个 PR：实现核心业务逻辑
3. 第三个 PR：添加 UI 和集成

这样更容易 Review，也更容易发现潜在问题。
```

## 最佳实践

### 作为 Reviewer

1. **及时响应**：在 24 小时内完成 Review
2. **友好礼貌**：使用建设性语言
3. **解释原因**：不仅说"怎么做"，还要说"为什么"
4. **承认学习**：如果不确定，说"我想了解一下"
5. **感谢贡献**：开始时说"感谢你的 PR"

### 作为 Author

1. **自审提交**：提交前自己先 Review 一遍
2. **小步快跑**：保持 PR 小而聚焦
3. **响应及时**：及时回复 Review 意见
4. **解释决定**：如果不同意建议，解释原因
5. **迭代改进**：根据反馈持续改进

### 沟通技巧

**好的说法**：
- "我建议..." / "或许可以考虑..."
- "我不太理解这部分逻辑，能否解释一下？"
- "这可能是一个 bug，你确认一下？"
- "👍 这部分写得很好！"

**避免的说法**：
- "你写错了" → "这里可能有问题"
- "必须这样改" → "建议改用这种方式"
- "这代码看不懂" → "这部分逻辑有点复杂，能否加些注释？"
